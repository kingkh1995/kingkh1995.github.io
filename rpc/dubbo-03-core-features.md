## 第9章 服务发现与负载均衡

### 9.1 服务发现的工作原理

Dubbo 的服务发现由 Registry 层统一管理。其核心工作流为：

1. **服务注册**：Provider 启动时，将自身的服务元数据（接口名、IP、端口、协议、配置等）注册到注册中心
2. **服务订阅**：Consumer 启动时，订阅目标服务，注册中心将当前可用的 Provider 列表推送给 Consumer
3. **变更通知**：当 Provider 宕机、扩容、缩容时，注册中心通过事件机制通知 Consumer
4. **动态更新**：Consumer 收到变更通知后，更新本地地址缓存，重建连接

**地址缓存机制**：Consumer 会缓存从注册中心获取的地址列表。当注册中心不可用时，缓存仍可支撑服务调用，但新上线或故障的 Provider 将无法被感知。

**服务发现配置框架**：通过 `dubbo.registry` 配置控制服务发现的行为，包括地址、协议、是否注册、是否订阅、分组等参数。

### 9.2 负载均衡策略

Dubbo 内置了 5 种负载均衡策略，通过 SPI 扩展机制可自定义策略。

#### 9.2.1 Random（加权随机）

**算法原理**：基于权重比率的随机选择。构建一个与权重总和等长的虚拟列表，通过均匀随机数选择目标。

**关键特性**：
- 支持权重的动态调整
- 在大流量下收敛于权重比例

**适用场景**：通用场景，当各 Provider 性能均衡时采用。

#### 9.2.2 RoundRobin（加权轮询）

**算法原理**：采用平滑加权轮询（Smooth Weighted Round-Robin）算法，避免短时间内的权重抖动。

**关键特性**：
- 每个周期内请求严格按权重比例分配
- 平滑算法确保请求在时间轴上均匀分布

**适用场景**：需要严格均分流量的场景。

#### 9.2.3 LeastActive（最少活跃数）

**算法原理**：Consumer 统计当前正在处理的请求数（活跃数），新请求优先分配给活跃数最少的 Provider。

**关键特性**：
- 自动将流量向处理能力强的服务倾斜
- 处理能力慢的 Provider 会自然获得更少流量
- 是动态自适应的策略

**适用场景**：Provider 性能差异较大的场景。

#### 9.2.4 ConsistentHash（一致性哈希）

**算法原理**：将请求的特定参数（默认第一个参数）进行哈希，映射到一个 2^32 的哈希环上，通过哈希环确定 Provider。

**关键特性**：
- 相同参数的请求总是分配到同一个 Provider
- Provider 增减时只影响少量请求的映射关系
- 支持虚拟节点（控制请求的集中程度）

**适用场景**：需要请求亲和性（会话保持）的场景。

#### 9.2.5 ShortestResponse（最短响应时间）

**算法原理**：记录每个 Provider 的历史响应时间，选择响应时间最短的 Provider 分配请求。响应时间越长获得的流量越少。

**关键特性**：
- 动态感知 Provider 性能变化
- 适合延迟敏感型应用

**适用场景**：延迟敏感的服务调用。

#### 9.2.6 选型对比

| 策略 | 加权支持 | 动态自适应 | 请求亲和性 | 推荐场景 |
|------|---------|-----------|-----------|---------|
| Random | 是 | 否 | 否 | 通用场景 |
| RoundRobin | 是（平滑） | 否 | 否 | 严格均分流量 |
| LeastActive | 否（但等价于加权） | 是 | 否 | 性能异构集群 |
| ConsistentHash | 否 | 否 | 是 | 需要会话保持 |
| ShortestResponse | 否 | 是 | 否 | 延迟敏感场景 |

## 第10章 流量管控

### 10.1 路由机制整体介绍

Dubbo 的路由机制允许在 Consumer 端对服务调用进行精细化控制，实现灰度发布、环境隔离、同区域优先等高级流量管理策略。

**架构层次**：路由规则在 Cluster 层的 Router 链中执行，先于负载均衡。所有合法的 Provider 地址经过 Router 链过滤后，再进入负载均衡阶段。

**路由规则来源**：
- 注册中心动态配置
- 本地配置文件
- 自定义 Router SPI 扩展

### 10.2 条件路由

条件路由基于特定的条件语法来筛选服务的提供者。

**路由规则结构**：
- `when` 条件：匹配 Consumer 的特征
- `then` 条件：筛选满足要求的 Provider
- 支持 IP、端口、方法名、参数等维度

**应用场景**：
- 只向特定 IP 段的 Provider 发送请求
- 针对特定服务方法的流量引导

### 10.3 标签路由

标签路由（Tag Router）是 Dubbo 最强大的流量管控手段之一，广泛应用于灰度发布和环境隔离。

**标签的定义与赋值**：
- Provider 端通过配置指定所属标签（如 `tag=gray`、`tag=production`）
- Consumer 端通过 RpcContext 指定要访问的标签

**灰度发布**：新版服务打上 `gray` 标签，仅部分内部用户流量导向灰度版本，验证通过后全量发布。

**多环境隔离**：开发、测试、预发布环境分别使用不同的标签，实现逻辑隔离而不需要物理隔离。

### 10.4 脚本路由

支持通过脚本语言（如 JavaScript、Groovy）编写路由逻辑，实现复杂的动态路由策略。

**适用场景**：需要编写复杂条件逻辑，规则变化频繁的场景。

### 10.5 动态配置规则

通过注册中心或配置中心动态下发配置规则，覆盖应用级别的配置。

**配置覆盖机制**：
- Consumer 端配置优先于 Provider 端配置
- 动态配置下发优先于本地配置
- 支持对特定服务、特定方法的覆盖

### 10.6 权重路由

通过给不同 Provider 设置不同的权重值，实现按比例分配流量。权重可以动态调整，无需重启服务。

### 10.7 同区域优先

在大规模部署中，跨机房调用会产生额外的网络延迟和带宽成本。同区域优先策略确保 Consumer 优先访问同机房/同 Region 的 Provider。

**工作原理**：
- Provider 注册时声明所属 Region
- Consumer 在选择时优先匹配同 Region 的 Provider
- 同 Region 无可用 Provider 时，Fallback 到其他 Region

### 10.8 服务降级

通过 Mock 策略实现服务降级，在 Provider 不可用或响应过慢时自动返回 Mock 结果。

**降级策略**：
- `force:return`：直接返回固定值，不发起远程调用
- `fail:return`：远程调用失败时返回固定值
- `force:throw`：直接抛出异常
- `fail:throw`：远程调用失败时抛出异常

**应用场景**：大促期间对弱依赖服务降级，确保核心链路不会被次要服务的故障影响。

## 第11章 限流与熔断

### 11.1 限流场景与需求分析

在分布式系统中，限流的核心目标是保护服务不被突发流量冲垮。限流可以在 Consumer 端（控制发送速率）和 Provider 端（控制处理速率）分别实施。

**核心需求**：
- 防止系统过载
- 确保核心链路的服务质量
- 控制资源消耗在预期范围内

### 11.2 Sentinel 集成

Sentinel 是阿里开源的流量控制组件，Dubbo 提供了与 Sentinel 的原生集成。

**规则定义**：
- **流量控制规则**：按 QPS 或线程数进行控制
- **熔断降级规则**：根据响应时间、异常比例、异常数触发熔断
- **系统保护规则**：根据系统负载（Load）自适应控制入口流量

**控制台**：
- Dubbo Admin 内置 Sentinel 控制面板
- 实时监控 QPS、RT、拒绝数等指标
- 在线修改限流规则

### 11.3 框架内置并发控制

Dubbo 框架内置了 `executes` 限制机制：

- `executes`：服务端每个服务方法的并发执行上限
- 超过上限的请求直接拒绝，返回异常
- 适合 Provider 端保护自身不被过多并发请求压垮

**线程池隔离**：不同服务或服务方法配置独立的线程池，避免某个慢服务耗尽所有线程资源。

### 11.4 自适应限流

Dubbo 3.x 提供了自适应限流算法，根据系统的实时状态动态调整限流阈值。

**算法原理**：
- 监控 CPU 使用率、响应时间、请求成功率
- 当系统处于健康状态时逐步提高限流上限
- 当系统负载升高时主动降低限流阈值

**优势**：无需人工配置精确的阈值，系统自行寻找最佳限流点。

### 11.5 熔断策略

熔断器（Circuit Breaker）用于防止级联故障：

**状态机**：
```
Closed（关闭） → Open（打开） → Half-Open（半开） → Closed（关闭）
```

- **Closed**：正常状态，所有请求正常通过
- **Open**：熔断触发，直接返回错误
- **Half-Open**：熔断超时后尝试放行少量请求，检查服务是否恢复
- **Closed**：请求成功，关闭熔断器

**触发条件**：
- 请求错误率达到阈值
- 响应时间超过阈值
- 慢调用比例过高

## 第12章 可观测性

### 12.1 可观测性的三大支柱

Dubbo 的可观测性体系由三大支柱构成：

| 支柱 | 说明 | 核心工具 |
|------|------|---------|

| Logging | 日志记录，追踪行为轨迹 | Dubbo AccessLog / Log4j |
| Tracing | 全链路追踪，关联分布式调用 | Zipkin / Skywalking / OTel |

### 12.2 可视化控制台

Dubbo Admin 是官方提供的可视化控制台，提供：

- **服务治理**：服务查询、服务详情、服务测试
- **配置管理**：动态配置下发、路由规则管理
- **监控面板**：Metrics 可视化
- **限流控制**：Sentinel 规则管理

### 12.3 Prometheus 监控指标

Dubbo 通过 Micrometer 暴露标准的 Prometheus 指标：

**内置指标列表**：

| 指标 | 类型 | 说明 |
|------|------|------|
| `dubbo_requests_total` | Counter | 请求总数 |
| `dubbo_request_duration_seconds` | Histogram | 请求耗时分布 |
| `dubbo_requests_active` | Gauge | 当前活跃请求数 |
| `dubbo_requests_error_total` | Counter | 请求错误数 |

### 12.4 Grafana 仪表盘

Dubbo 社区提供了预置的 Grafana Dashboard：

- 全局概览面板：QPS、错误率、P99 延迟

- Provider/Consumer 面板：分角色监控

### 12.5 全链路追踪

**追踪原理**：在 RPC 调用中，Tracing 系统通过 Trace ID 和 Span ID 将分布式调用串联为一条追踪链。

**支持的集成方案**：

| 方案 | 协议 | 适合场景 |
|------|------|---------|

| Zipkin | HTTP/Thrift | 传统方案，与 Zipkin 生态集成 |
| Skywalking | gRPC | 需要额外分析能力的场景 |

## 第13章 安全策略

### 13.1 安全威胁模型

Dubbo 服务在微服务架构中面临的主要安全威胁：

| 威胁 | 风险描述 |
|------|---------|
| 网络窃听 | 未加密的传输数据可以被中间人窃取 |
| 未授权访问 | 任意 Consumer 可以调用未保护的服务 |
| 反序列化攻击 | 恶意序列化数据可能导致远程代码执行 |
| 重放攻击 | 捕获的合法请求可能被重复发送 |

### 13.2 TLS 传输加密

Dubbo 支持基于 TLS 的传输加密：

- **单向 TLS**：Consumer 验证 Provider 证书，确保连接到正确的服务
- **双向 TLS（mTLS）**：Provider 和 Consumer 互相验证证书
- **证书管理**：支持 JKS/PEM 格式，支持证书热加载

在服务网格中，mTLS 通常由 Istio 或 Envoy 自动管理。

### 13.3 Token 鉴权

Dubbo 提供简单的 Token 鉴权机制：

- Provider 端配置 Token 值
- Consumer 端需在请求中携带匹配的 Token
- 不匹配的请求将被 Provider 拒绝

**适用场景**：同一网络内的简单访问控制。

### 13.4 服务认证鉴权

Dubbo 3.x 提供了更全面的服务认证鉴权能力：

- 基于 Access Key ID / Secret Access Key 的认证
- 支持自定义鉴权扩展
- 鉴权失败的请求被拦截并返回异常

### 13.5 类检查机制

**安全背景**：Java 反序列化漏洞是 Dubbo 历史上最严重的安全威胁。Dubbo 3.x 引入了类检查机制。

**工作方式**：
- 服务端在反序列化时检查目标类是否在白名单中
- 白名单外的类反序列化将被拦截
- 支持自定义白名单/黑名单配置

## 第14章 序列化

### 14.1 Dubbo 序列化体系架构

Dubbo 的序列化通过 Serialization SPI 抽象层实现：

```
Serialization（序列化接口）
  ├── ObjectInput    [反序列化输入]
  └── ObjectOutput   [序列化输出]

实现：
  ├── Hessian2Serialization（默认）
  ├── ProtobufSerialization
  ├── Fastjson2Serialization
  ├── KryoSerialization
  ├── FstSerialization
  ├── AvroSerialization
  ├── GsonSerialization
  ├── MessagePackSerialization
  └── JavaSerialization（不推荐）
```

序列化方式与传输协议耦合：triple 协议默认使用 Protobuf，dubbo 协议默认使用 Hessian2。

### 14.2 triple 协议序列化

**Protobuf（Protocol Buffers）**：Google 开源的跨语言序列化方案。强类型约束，需预定义 `.proto` 文件。性能优于 JSON，数据体积小。

**Protobuf Wrapper 模式**：在 Java 接口模式下无需编写 `.proto` 文件，底层自动将 Java POJO 转换为 Protobuf 格式。Dubbo 自动生成 Protobuf 映射描述。适合 Java 技术栈的场景，保持开发效率。

**JSON 序列化**：在 triple 协议下支持 JSON 格式，适合调试和非严格类型的场景。

### 14.3 dubbo 协议序列化

| 序列化 | 类型 | 特点 | 适用场景 |
|--------|------|------|---------|
| Hessian2 | 二进制 | 默认序列化，跨语言支持 | 通用场景 |
| Fastjson2 | JSON | 高性能，兼容性优于 Fastjson | 需要 JSON 格式 |
| Kryo | 二进制 | 极致性能，Java 原生 | 纯 Java 内网调用 |
| FST | 二进制 | 类似 Kryo，性能优异 | 纯 Java 内网调用 |
| Avro | 二进制 | Hadoop 生态兼容 | 大数据场景 |
| Gson | JSON | Google 出品，功能全面 | 结构化数据场景 |
| MessagePack | 二进制 | 类 JSON 二进制 | 数据体积敏感场景 |

### 14.4 序列化性能对比（参考）

| 序列化 | 编码速度 | 解码速度 | 数据体积 | 跨语言 |
|--------|---------|---------|---------|-------|
| Kryo | ★★★★★ | ★★★★★ | ★★★★★ | ★ |
| FST | ★★★★★ | ★★★★★ | ★★★★★ | ★ |
| Hessian2 | ★★★★ | ★★★★ | ★★★★ | ★★★★ |

| Avro | ★★★ | ★★★ | ★★★★ | ★★★★★ |

### 14.5 序列化升级策略

在升级序列化方式时需考虑兼容性：

1. **兼容性测试**：确保新序列化能够处理已有数据的序列化和反序列化
2. **灰度升级**：先升级一部分节点验证兼容性
3. **双读写**：在迁移期同时支持新旧两种格式
4. **回滚方案**：确保升级失败时可以快速切回原有序列化方式
