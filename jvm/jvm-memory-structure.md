## 运行时数据区与对象内存布局

JVM 的内存管理是每一位后端工程师必须掌握的底层知识。理解运行时数据区的划分逻辑、对象在内存中的具体布局，以及各区域在生产环境中常见的 OutOfMemoryError（OOM）场景，是排查线上内存问题、优化系统吞吐量的基础。

本文聚焦 HotSpot VM，从实战视角梳理 JVM 内存结构的完整图景，涵盖运行时数据区、对象内存布局、JDK 版本演进带来的关键变化，以及各区域 OOM 的排查思路。内容面向有一定经验的资深后端开发，不过度纠缠于位运算和内存地址计算，而是聚焦生产环境中的实际问题和排查方法。

***

## 运行时数据区

JVM 在执行 Java 程序时，会将内存划分为若干功能各异的数据区域。这些区域有的在虚拟机启动时创建，有的随线程生命周期而创建和销毁；有的被所有线程共享，有的则是线程私有的。

从宏观上看，运行时数据区可以分为两大类：

- **线程私有区域**：程序计数器、Java 虚拟机栈、本地方法栈。这些区域的生命周期与线程绑定，线程创建时分配，线程结束时销毁。
- **线程共享区域**：Java 堆、方法区、直接内存。这些区域在虚拟机启动时创建，被所有线程共享，是垃圾收集器管理和 OOM 发生的主要场所。

> 理解各区域的职责边界和共享属性，是定位内存问题的第一步。当线上出现内存异常时，首先要判断异常发生在哪个区域，然后再针对性地深入分析。

***

### 程序计数器

程序计数器（Program Counter Register）是一块较小的内存空间，为**线程私有**。它用于记录当前线程正在执行的字节码指令的地址。如果当前线程正在执行一个 Native 方法，则程序计数器的值为 Undefined。

程序计数器的主要作用包括：

- **线程切换恢复**：多线程环境下，CPU 时间片轮转后需要依赖程序计数器恢复到正确的执行位置。每个线程都有自己独立的程序计数器，互不影响。
- **分支与跳转**：字节码解释器通过修改程序计数器的值来实现分支、循环、跳转等控制流。程序计数器的值指向下一条将要执行的指令地址。
- **异常处理**：在异常发生时，程序计数器帮助虚拟机定位异常发生的位置，以便生成准确的堆栈跟踪信息。

**程序计数器是唯一不会发生 OOM 的区域**，因为它所占用的内存空间在编译期就已完全确定，且每个线程只有一个程序计数器，不会因为运行时数据的增长而扩展。

在实际开发中，程序计数器对开发者是透明的，我们不需要也无法直接操作它。但在分析多线程问题时，理解程序计数器的存在有助于理解为什么线程切换不会导致执行位置错乱——每个线程都有自己的"执行书签"。

### Java 虚拟机栈

Java 虚拟机栈（JVM Stack）同样是**线程私有**的，描述的是 Java 方法执行的内存模型。每个方法被执行时，JVM 都会同步创建一个**栈帧**（Stack Frame），用于存储方法的局部变量、操作数、动态链接和方法返回地址等信息。方法执行完毕后，栈帧出栈销毁。

虚拟机栈的大小可以通过 `-Xss` 参数设置。如果线程请求的栈深度超过虚拟机允许的最大深度，会抛出 `StackOverflowError`；如果虚拟机栈允许动态扩展（HotSpot 不支持），扩展时无法申请到足够内存会抛出 `OutOfMemoryError`。

#### 栈帧结构

一个完整的栈帧包含以下四个部分，每一部分都在方法执行过程中扮演着不可或缺的角色：

| 区域 | 说明 | 实战关注点 |
|------|------|-----------|
| 局部变量表 | 存放方法参数和方法内部定义的局部变量 | 基本数据类型、对象引用和 returnAddress；long / double 占两个变量槽 |
| 操作数栈 | 方法执行过程中的工作区，用于计算和传参 | 最大深度在编译期确定 |
| 动态链接 | 指向运行时常量池中当前方法的引用 | 支持运行期符号引用到直接引用的动态解析 |
| 方法返回地址 | 方法退出后恢复主调方法的执行位置 | 正常返回时由程序计数器提供，异常返回时由异常表决定 |

**局部变量表**以**变量槽**（Slot）为基本单位，一个 Slot 可以存放 32 位以内的数据类型。boolean、byte、char、short、int、float、reference 和 returnAddress 各占一个 Slot；long 和 double 由于占用 64 位，需要两个连续的 Slot。

需要特别注意的是，**局部变量表中的对象引用是重要的 GC Roots**。当方法还在执行时，局部变量表里的引用会阻止对应对象被垃圾回收。这也意味着，如果在方法中持有一个大对象的引用，即使后续代码不再使用该对象，只要方法未结束且局部变量表未被覆盖，该对象就无法被回收。

> 一个常见的优化技巧是：在方法中如果某个大对象不再需要，可以显式将其引用置为 `null`，帮助 GC 提前回收。不过在现代 JVM 中，JIT 编译器的逃逸分析和死代码消除通常能自动处理这种情况，手动置 null 的收益已经不如早期 JDK 明显。

**操作数栈**是方法执行过程中的工作区。字节码指令通过向操作数栈压入和弹出数据来完成计算。例如，执行 `iadd` 指令时，会从操作数栈顶弹出两个 int 值，相加后将结果压回栈顶。操作数栈的最大深度在编译期就已经确定，并记录在 Class 文件的 Code 属性中。

**动态链接**将符号引用转换为直接引用。每个栈帧都包含一个指向运行时常量池中该栈帧所属方法的引用，目的是支持方法调用过程中的动态解析。被 `invokedynamic` 指令调用的方法（如 Lambda 表达式）尤其依赖这一机制。

**方法返回地址**用于在方法退出时恢复到主调方法的正确执行位置。方法退出的方式有两种：正常返回（遇到返回指令）和异常返回（遇到未捕获的异常）。正常返回时，程序计数器的值可以作为返回地址；异常返回时，需要通过异常处理表来确定返回地址。

#### 栈溢出实战排查

虚拟机栈可能出现的异常有两种，排查思路截然不同：

**StackOverflowError**：

- **触发原因**：线程请求的栈深度超过虚拟机允许的最大深度。最常见的原因是无限递归或递归深度过大。另一个容易被忽视的场景是对象间的循环依赖导致的相互调用，比如两个对象的 `toString()` 方法相互引用。
- **典型场景**：深度优先搜索（DFS）未对递归深度做限制、复杂的对象关系图遍历、ORM 框架中关联对象的双向引用未做懒加载处理。
- **排查方法**：查看堆栈跟踪，通常最后几行就是递归调用的入口。检查是否存在循环依赖，或者递归终止条件是否正确。

**OutOfMemoryError: unable to create new native thread**：

- **触发原因**：创建线程时操作系统无法分配足够的内存。HotSpot 中虚拟机栈和本地方法栈合并实现，不支持动态扩展，因此栈区域的 OOM 通常表现为无法创建新线程，而非栈扩展失败。
- **典型场景**：线程池配置不合理（如 `maximumPoolSize` 设置过大）、线程泄漏（线程创建后未正确回收）、定时任务框架配置错误导致线程数量激增。
- **排查方法**：
  1. 使用 `jstack <pid>` 查看当前线程数量和线程状态。
  2. 检查线程池配置，确认最大线程数是否合理。
  3. 检查是否存在线程泄漏（如线程池中的线程因阻塞而未被回收）。
  4. 检查操作系统对进程线程数的限制（Linux 的 `ulimit -u`）。
  5. 适当减小 `-Xss` 参数（注意：单个线程栈内存减小，允许创建的总线程数会增加，但单个线程的栈深度会减少）。

> **HotSpot 虚拟机不区分虚拟机栈和本地方法栈，两者共用 `-Xss` 参数设置栈大小，且不支持动态扩展。** 这是 HotSpot 的设计选择，简化了栈管理的复杂度。

### 本地方法栈

本地方法栈（Native Method Stack）与虚拟机栈的作用非常相似，区别在于它是为 Native 方法服务的。在 HotSpot 中，本地方法栈与虚拟机栈被合并实现，因此两者的行为和参数完全一致。

Native 方法是使用 Java 以外的语言（如 C / C++）编写的方法，通过 JNI（Java Native Interface）与 Java 代码交互。本地方法栈同样会发生 `StackOverflowError` 和 `OutOfMemoryError`，只是触发场景与 Native 代码的调用深度和线程创建相关。

在实际开发中，本地方法栈的问题通常与第三方 Native 库（如 Netty 的 Native 传输、RocketMQ 的存储引擎）相关。如果怀疑本地方法栈出现问题，可以结合 Native 库的日志和系统日志（如 Linux 的 `dmesg`）进行排查。

### Java 堆

Java 堆（Heap）是 JVM 管理的最大一块内存区域，**被所有线程共享**，在虚拟机启动时即创建。它的唯一目的就是存放对象实例和数组，是垃圾收集器管理的主要区域。

从垃圾收集的角度来看，现代 JVM 通常将堆划分为**新生代**（Young Generation）和**老年代**（Old Generation），新生代又进一步划分为 **Eden 区**和两个 **Survivor 区**（From 和 To）。这种分代设计基于**弱分代假设**：绝大多数对象都是朝生夕灭的，熬过越多次垃圾收集的对象就越难以消亡。虽然本文不展开讲解 GC 算法的细节，但理解堆的分代结构有助于理解对象在堆中的生命周期。

堆的大小可以通过 `-Xms`（初始大小）和 `-Xmx`（最大大小）参数设定。当堆中没有足够内存完成实例分配，且无法继续扩展时，会抛出 `OutOfMemoryError: Java heap space`。

需要澄清一个常见的误解：**"所有的对象实例都在堆上分配"并不绝对**。现代 JVM 通过逃逸分析（Escape Analysis）可以进行栈上分配、标量替换等优化，将部分对象分配在栈上而非堆上，从而减少 GC 压力。

#### TLAB 与高并发分配

TLAB（Thread Local Allocation Buffer，线程本地分配缓冲区）是 Java 堆中**线程私有**的分配区域。它的存在是为了解决多线程并发环境下对象分配的性能瓶颈。

在多线程高并发场景下，如果所有线程都在同一块堆内存上进行对象分配，必然需要同步机制（如 CAS 操作）来保证线程安全。虽然单次 CAS 操作的开销很小，但在高并发、小对象频繁创建的场景下，这种同步竞争会显著降低对象分配的效率，成为系统瓶颈。

TLAB 的设计思路非常简洁有效：**为每个线程预先在堆中的 Eden 区划分一小块私有内存，线程优先在自己的 TLAB 上分配对象，只有当 TLAB 用完时才需要同步申请新的缓冲区。**

TLAB 带来的实战价值：

- **消除竞争**：绝大多数对象分配不需要同步，大幅提升对象分配效率。在高并发场景下，TLAB 可以将对象分配的性能提升数倍。
- **缓存友好**：TLAB 空间连续，有利于 CPU 缓存命中。连续分配的对象在内存中相邻，访问时缓存命中率更高。
- **默认开启**：在 HotSpot 中，TLAB 默认是开启的，可以通过 `-XX:+/-UseTLAB` 控制。在绝大多数场景下，保持默认开启是最优选择。

> 在高并发、小对象频繁创建的系统（如网关、RPC 框架、消息中间件）中，TLAB 的效果尤为明显。如果观察到对象分配成为瓶颈，可以检查 `-XX:TLABSize` 参数是否设置合理。TLAB 过小会导致线程频繁申请新缓冲区，增加同步开销；TLAB 过大则会浪费 Eden 区空间，降低内存利用率。

**TLAB 的局限性**：TLAB 只适用于小对象的快速分配。如果对象大小超过 TLAB 的剩余空间（或者超过 TLAB 大小的一半），JVM 会直接在 Eden 区中分配（可能需要同步），而不是分割 TLAB。这也是为什么在某些极端场景下，大对象的创建会成为并发瓶颈。

#### 逃逸分析与栈上分配

逃逸分析是 JIT 编译器的一项优化技术，用于判断对象的作用域是否逃逸出当前方法或线程。基于逃逸分析的结果，JVM 可以进行多种优化：

- **栈上分配**（Stack Allocation）：如果对象没有发生方法逃逸（即对象只在当前方法中使用，不会被返回或传递给其他方法），JVM 可以将对象直接分配在栈帧中，方法结束后随栈帧销毁而回收，无需 GC 介入。
- **标量替换**（Scalar Replacement）：如果对象没有发生方法逃逸，且对象的字段可以被分解为基本类型，JIT 编译器可能不创建对象，而是直接使用对象的字段作为局部变量。
- **锁消除**（Lock Elision）：如果对象没有发生线程逃逸（即对象不会被其他线程访问），JVM 可以安全地去掉该对象上的同步锁。

逃逸分析的判定维度：

- **方法逃逸**：对象被作为参数传递给其他方法，或作为返回值返回。
- **线程逃逸**：对象被其他线程访问（如赋值给静态变量、存储到共享集合中）。

> 逃逸分析默认开启（`-XX:+DoEscapeAnalysis`），但在解释执行阶段不生效，只有在 JIT 编译后的代码中才能体现优化效果。这也是为什么某些微基准测试结果与实际运行存在差异的原因之一——微基准可能运行在解释模式下，未能触发逃逸分析优化。

在生产环境中，如果代码中存在大量短生命周期的临时对象（如在循环中创建的临时对象），且这些对象不会逃逸出方法，栈上分配可以显著减少 GC 的频率和压力。但需要注意的是，栈上分配的对象无法被 GC 跟踪，因此如果对象发生了方法逃逸而 JIT 未能正确识别，可能会导致严重的内存安全问题。不过现代 JVM 的逃逸分析算法已经非常成熟，这种情况极少发生。

### 方法区

方法区（Method Area）是**被所有线程共享**的内存区域，用于存储已被加载的类信息、常量、静态变量、即时编译器编译后的代码缓存等数据。方法区在逻辑上属于堆的一部分，但物理上通常与 Java 堆隔离，因此也被称为**"非堆"**（Non-Heap）。

方法区的垃圾收集目标主要是**废弃常量的回收**和**不再使用的类型卸载**。当方法区无法满足新的内存分配需求时，会抛出 OOM。不过方法区的垃圾收集效果通常不如堆，尤其是类型卸载的条件非常苛刻。

#### 永久代到元空间的演变

在 JDK 8 之前，HotSpot 使用**永久代**（Permanent Generation，PermGen）实现方法区。永久代使用的是 JVM 自身的内存空间，其大小由 `-XX:MaxPermSize` 参数限制，默认值通常较小（如 64 MB 或 128 MB）。

永久代在生产环境中的痛点非常突出：

- **OOM 频繁**：动态代理、反射、字节码增强（如 CGLIB、ASM、Javassist）会大量生成类，极易撑满永久代。Spring、Hibernate、MyBatis 等框架在复杂场景下经常触发 `OutOfMemoryError: PermGen space`。
- **大小难以预估**：永久代大小需要在启动时固定，过大浪费内存，过小频繁 OOM。对于动态类加载密集型的应用，永久代的大小几乎无法准确预估。
- **GC 效率低**：永久代的垃圾收集效果不佳，类型卸载的条件苛刻（需要该类所有实例都已被回收、加载该类的类加载器已经被回收、该类的 Class 对象没有在任何地方被引用）。
- **与堆内存竞争**：永久代与 Java 堆共享 JVM 内存空间，永久代的增长会压缩堆的可用空间，反之亦然。

从 JDK 8 开始，HotSpot 移除了永久代，改用**元空间**（Metaspace）实现方法区。元空间使用的是**本地内存**（Native Memory），不再受 JVM 堆内存的限制，只受物理内存和操作系统进程内存上限的约束。

| 特性 | 永久代（PermGen） | 元空间（Metaspace） |
|------|------------------|-------------------|
| 内存位置 | JVM 堆内存的一部分 | 本地内存（Native Memory） |
| 大小限制 | `-XX:MaxPermSize`（默认较小，如 64 MB） | `-XX:MaxMetaspaceSize`（默认无限制） |
| 字符串常量池 | 位于永久代 | 移入 Java 堆 |
| 静态变量 | 位于永久代 | 移入 Java 堆 |
| 垃圾收集 | Full GC 时清理，效率低 | 独立触发，更灵活 |
| 与堆的关系 | 与堆竞争内存 | 独立管理，互不干扰 |

> **为什么方法区要从永久代改为元空间？** 核心原因是解决生产环境中动态类加载导致的 OOM 问题，同时让方法区的大小能够根据实际使用情况动态伸缩，避免固定上限带来的运维困扰。Oracle 官方给出的另一个原因是，将方法区移入本地内存后，可以与 JRockit VM 的实现保持一致，为后续融合打下基础。

元空间的关键参数：

- `-XX:MaxMetaspaceSize`：设置元空间最大值，默认 `-1` 表示不限制，只受物理内存约束。在生产环境中，建议设置一个合理的上限，防止元空间无限制增长导致系统 OOM Killer 介入。
- `-XX:MetaspaceSize`：元空间初始阈值，达到该值会触发垃圾收集进行类型卸载，同时 GC 会动态调整该值。如果设置得过小，会导致频繁的元空间 GC；如果设置得过大，会延迟首次 GC 的时机。
- `-XX:MinMetaspaceFreeRatio / -XX:MaxMetaspaceFreeRatio`：GC 后元空间空闲容量的目标百分比区间，用于动态调整元空间大小。空闲空间过大会造成内存浪费，过小又会增加 GC 频率。

#### 运行时常量池

运行时常量池（Runtime Constant Pool）是方法区的一部分。Class 文件中的常量池在类加载后进入方法区，成为运行时常量池。它不仅存放编译期生成的字面量和符号引用，还具有**动态性**——运行期间可以将新的常量放入池中，最典型的例子是 `String.intern()` 方法。

运行时常量池的动态性带来了灵活性，但也可能带来内存问题。如果代码中大量调用 `String.intern()`，且 intern 的字符串没有复用价值，运行时常量池可能会不断增长，最终触发方法区 OOM（在 JDK 7 之前）或堆 OOM（在 JDK 7 及之后）。

> 从 JDK 7 开始，**字符串常量池被移入 Java 堆**。这意味着大量使用 `intern()` 的场景需要关注堆内存而非方法区内存。这一改动也是永久代移除的前奏——字符串常量池是永久代中占用空间较大的部分之一。

### 直接内存

直接内存（Direct Memory）**不属于 JVM 运行时数据区**，也不在 Java 虚拟机规范中定义。它是在 NIO（New Input/Output）类中引入的，通过 `DirectByteBuffer` 对象操作 Native 函数直接分配的堆外内存。

直接内存的主要用途和优势：

- **零拷贝**：在进行 I/O 操作时（如网络传输、文件读写），直接内存可以避免数据在 Java 堆和 Native 堆之间的来回复制，显著提升 I/O 性能。Netty、Kafka 等高性能框架大量使用直接内存来实现零拷贝。
- **大对象存储**：超出堆内存限制的大对象可以存放在直接内存中，避免堆内存被大对象占满导致 GC 压力过大。
- **减少 GC 压力**：直接内存不受 JVM 堆 GC 管理，不会因为频繁创建和销毁大对象而触发频繁的 GC。

直接内存的大小可以通过 `-XX:MaxDirectMemorySize` 参数指定，不指定则默认与 Java 堆最大值（`-Xmx`）一致。

`DirectByteBuffer` 对象的回收依赖关联的 Cleaner 机制：当 `DirectByteBuffer` 对象被 GC 回收（无论是 Minor GC 还是 Full GC）后，其 Cleaner 会被加入 ReferenceQueue，由 ReferenceHandler 线程异步释放对应的直接内存。如果 `DirectByteBuffer` 分配速度远快于 GC 回收速度，可能导致 `OutOfMemoryError: Direct buffer memory`。

**排查思路**：

1. 检查是否大量使用 NIO 或 Netty 等依赖直接内存的框架。可以通过 `jcmd <pid> VM.native_memory` 查看 Native Memory 的详细使用情况。
2. 监控直接内存使用量。直接内存的使用量不会直接显示在堆内存监控中，需要通过 Native Memory Tracking（NMT）来查看。
3. 检查代码中 `DirectByteBuffer` 的创建位置，是否存在创建后未合理释放的情况。虽然 `DirectByteBuffer` 依赖 GC 回收后触发 Cleaner，但在高并发场景下，如果 `DirectByteBuffer` 对象被频繁创建且短暂存活，可能导致 Cleaner 队列堆积。
4. 适当增大 `-XX:MaxDirectMemorySize` 参数，或优化代码中 `DirectByteBuffer` 的创建和释放逻辑。在极端情况下，可以考虑显式调用 `((DirectBuffer) buffer).cleaner().clean()` 来主动释放直接内存，但这种方式需要谨慎使用，因为可能干扰 GC 的正常行为。

> 启用 Native Memory Tracking 的方法是添加 JVM 参数 `-XX:NativeMemoryTracking=summary`（或 `detail`），然后通过 `jcmd <pid> VM.native_memory summary` 查看。注意 NMT 本身会带来约 5% ~ 10% 的性能开销，不建议在生产环境长期开启。

## 对象的内存布局

了解 JVM 如何在堆中组织和访问对象，对于理解锁机制、GC 行为以及内存对齐对性能的影响至关重要。一个 Java 对象在堆中并非只是实例数据的简单堆积，而是有着严格的结构化布局。

### 对象的创建过程

当执行 `new` 指令时，JVM 会依次完成以下步骤，每一步都可能成为性能优化的切入点：

**1. 类加载检查**

虚拟机遇到 `new` 指令后，首先定位常量池中该类的符号引用，检查这个类是否已被加载、解析和初始化。如果没有，则必须先执行相应的类加载过程。类加载完成后，虚拟机才能确定对象需要分配的内存大小。

**2. 分配内存**

类加载检查通过后，为新对象分配内存。对象所需内存的大小在类加载完成后即可完全确定。内存分配方式有两种：

- **指针碰撞**（Bump the Pointer）：假设 Java 堆中的内存是绝对规整的，所有已使用的内存放在一边，空闲内存放在另一边，中间放着一个指针作为分界点。分配内存时，只需将指针向空闲空间方向移动一段与对象大小相等的距离。这种方式简单高效，但要求堆内存是规整的。
- **空闲列表**（Free List）：如果 Java 堆中的内存不是规整的，已使用的内存和空闲内存相互交错，虚拟机需要维护一个列表，记录哪些内存块是可用的。分配内存时，从列表中找到一块足够大的空间分配给对象，并更新列表。

选择哪种分配方式取决于垃圾收集器是否带有**空间压缩整理**（Compact）的能力。Serial、ParNew 等带有压缩过程的收集器使用指针碰撞；CMS 这种基于清除（Sweep）算法的收集器使用空闲列表。

**3. 内存初始化**

内存分配完成后，虚拟机需要将分配到的内存空间初始化为零值（不包括对象头）。这一步保证了对象的实例字段在 Java 代码中不赋初始值就能直接使用，对应类型的零值（如 int 为 0，boolean 为 false，引用类型为 null）。

这一步可以**提前到 TLAB 分配阶段**完成。当 TLAB 被初始化时，整个 TLAB 空间会被清零，后续在 TLAB 上分配对象时无需再次初始化。

**4. 对象头设置**

虚拟机对对象头进行必要的设置，包括：

- 类的元数据地址（类型指针）
- 对象的哈希码（通常延迟到第一次调用 `hashCode()` 时生成）
- GC 分代年龄（初始为 0）
- 锁状态标志（初始为无锁状态）
- 偏向线程 ID 和偏向时间戳（如果偏向锁开启）

**5. 执行构造方法**

在虚拟机看来，`new` 指令执行完成后对象已经创建。但从 Java 程序的角度看，对象的创建还没有完成——接下来会执行 `<init>()` 方法（即构造方法），按照程序员的意愿进行初始化。构造方法可能调用父类的构造方法、执行实例代码块和字段初始化表达式。

> 对象创建过程中的线程安全问题通过两种方式解决：一是采用 CAS 操作配合失败重试保证原子性；二是使用 TLAB 在线程私有空间分配，避免同步。在高并发场景下，TLAB 是对象分配的主要路径，只有在 TLAB 空间不足时才退回到同步分配。

### 对象头的实战意义

对象在堆中的内存布局分为三个部分：**对象头**（Header）、**实例数据**（Instance Data）和**对齐填充**（Padding）。其中对象头是最复杂也最具实战价值的部分。

#### Mark Word 与锁状态位

Mark Word 是对象头的第一部分，用于存储对象自身的运行时数据。在 64 位虚拟机中，Mark Word 占 64 位；在 32 位虚拟机中占 32 位。Mark Word 是一个**非固定数据结构**，会根据对象的状态复用自己的存储空间，以在极小的空间内存储尽可能多的信息。

在 64 位 JVM 中，Mark Word 的典型布局如下（无压缩指针，无压缩对象头）：

| 锁状态 | 存储内容 |
|--------|---------|
| 无锁 | 哈希码（31 bit）+ 分代年龄（4 bit）+ 偏向模式（1 bit）+ 锁标志（01） |
| 偏向锁 | 线程 ID（54 bit）+ Epoch（2 bit）+ 分代年龄（4 bit）+ 偏向模式（1 bit）+ 锁标志（01） |
| 轻量级锁 | 指向栈中锁记录的指针（62 bit）+ 锁标志（00） |
| 重量级锁 | 指向互斥量（Monitor）的指针（62 bit）+ 锁标志（10） |
| GC 标记 | 空（62 bit）+ 锁标志（11） |

对象头中的锁状态位与并发编程的关系极为密切。理解 Mark Word 的变化过程，就能理解 `synchronized` 锁升级的全过程：

**偏向锁**：

当第一个线程访问同步块时，JVM 会在 Mark Word 中记录该线程 ID。后续该线程再次进入同步块时，只需检查线程 ID 是否一致，无需任何同步操作，开销接近于零。偏向锁的假设是：在大多数情况下，锁不仅不存在多线程竞争，而且总是由同一线程多次获得。

**轻量级锁**：

当第二个线程尝试获取锁时，偏向锁被撤销，升级为轻量级锁。线程在自己的栈帧中创建锁记录（Lock Record），通过 CAS 操作尝试将 Mark Word 更新为指向锁记录的指针。如果 CAS 成功，线程获得锁；如果失败，说明存在竞争，进入自旋等待。

**重量级锁**：

如果自旋等待失败（通常自旋次数达到阈值），锁膨胀为重量级锁。此时 Mark Word 指向一个 Monitor 对象，线程进入阻塞状态，由操作系统调度。重量级锁的开销最大，涉及用户态和内核态的切换。

> **为什么偏向锁在 JDK 15 中被标记为废弃，在 JDK 21 中被彻底移除？** 因为现代应用普遍使用线程池，偏向锁的"同一个线程反复获取锁"假设在大多数场景下不成立。偏向锁的撤销和升级本身就有开销，在竞争激烈的情况下反而拖慢性能。JEP 374 的实验数据表明，禁用偏向锁后，许多工作负载的性能有所提升。移除偏向锁还简化了 JVM 的锁机制代码，降低了维护复杂度。
> 
> 在 **JDK 21+** 环境中，以下偏向锁相关内容仅作为历史参考，实际运行时偏向锁已不存在。

对象头中的锁状态位不仅影响 `synchronized` 的性能，还影响 GC 的行为。例如，在 GC 过程中，如果对象处于重量级锁状态，GC 需要额外的处理来确保锁的一致性。此外，对象的哈希码也与锁状态相关：一旦对象计算过哈希码（即调用了未重写的 `hashCode()`），该对象就无法进入偏向锁状态，因为哈希码和偏向线程 ID 共享同一段 Mark Word 空间。

> 这意味着，如果对象的 `hashCode()` 被提前调用（如放入 HashMap 中），该对象将永远失去偏向锁的资格。在高并发场景下，这可能导致原本可以通过偏向锁优化的同步块退化为轻量级锁或重量级锁，增加同步开销。

#### 类型指针与数组长度

对象头的第二部分是**类型指针**（Class Pointer），即指向对象所属类的元数据（Klass）的指针。通过类型指针，JVM 能够确定对象是哪个类的实例，从而在方法调用时进行动态分派（Dynamic Dispatch）。类型指针还用于获取对象的方法表（Method Table），支持虚方法调用。

在 64 位 JVM 中，类型指针默认也是 64 位。但通过 **指针压缩**（Compressed Oops，`-XX:+UseCompressedOops`）技术，可以将其压缩为 32 位，从而节省对象头空间。当堆内存小于 32 GB 时，指针压缩默认开启。指针压缩将 64 位地址映射到 32 位偏移量，通过基地址加偏移的方式访问对象，在节省内存的同时几乎不损失性能。

如果对象是一个**数组**，对象头中还会额外存储一个**数组长度**字段（32 位）。这意味着数组对象的对象头比普通对象多 4 字节（未压缩时）或 4 字节（压缩时，因为长度字段本身不参与压缩）。数组长度字段的存在使得 JVM 可以在 O(1) 时间内获取数组长度，而不需要遍历元数据。

> 对于包含大量小对象的系统，对象头本身占用的空间不容忽视。在 64 位 JVM 开启指针压缩的情况下，普通对象的对象头占用 12 字节（8 字节 Mark Word + 4 字节压缩类型指针），数组对象占用 16 字节（额外 4 字节数组长度）。如果系统中存在数千万个小对象，对象头的总空间可能达到数百 MB。

### 对象头结构

在深入 Mark Word 的锁状态变化之前，有必要先理清标准 HotSpot 对象头的整体结构。对象头并非单一字段，而是由多个固定宽度的片段组合而成，其总大小直接影响每个对象的内存开销。

在 64 位 JVM 上，标准对象头包含以下三部分：

| 片段 | 宽度 | 说明 |
|------|------|------|
| Mark Word | 64 bit | 存储对象运行时数据：哈希码、GC 分代年龄、锁状态标志、偏向线程 ID 等 |
| Class Pointer | 32 / 64 bit | 指向对象所属 Klass 的元数据地址；开启压缩指针（Compressed Oops）时为 32 bit，关闭时为 64 bit |
| Array Length | 32 bit | 仅数组对象存在，记录数组长度，使 JVM 可在 O(1) 时间获取数组大小 |

其中 **Mark Word** 是对象头的核心，占固定的 64 位（64 位 JVM）。它的内容会根据对象状态动态复用：

- **无锁状态**：存储哈希码（31 bit）+ GC 分代年龄（4 bit）+ 偏向模式（1 bit）+ 锁标志（2 bit）
- **偏向锁状态**：存储偏向线程 ID（54 bit）+ Epoch（2 bit）+ GC 分代年龄（4 bit）+ 偏向模式（1 bit）+ 锁标志（2 bit）
- **轻量级锁状态**：存储指向栈中锁记录的指针（62 bit）+ 锁标志（2 bit）
- **重量级锁状态**：存储指向 Monitor 的指针（62 bit）+ 锁标志（2 bit）
- **GC 标记状态**：存储 GC 标记信息（62 bit）+ 锁标志（2 bit）

**典型对象头大小**：

| 场景 | 普通对象头 | 数组对象头 |
|------|-----------|-----------|
| 64 位 JVM + 压缩指针（默认，堆 < 32 GB） | 12 字节（8 + 4） | 16 字节（8 + 4 + 4） |
| 64 位 JVM + 未压缩指针 | 16 字节（8 + 8） | 24 字节（8 + 8 + 4 + 对齐 4） |

> 由于 HotSpot 要求对象大小必须是 8 字节的整数倍，未压缩指针场景下数组对象头为 24 字节（16 字节基础头 + 4 字节数组长度 + 4 字节对齐填充）。对于包含数千万小对象的系统，对象头本身可能占据数百 MB 的堆空间，这也是 JDK 25 引入紧凑对象头的直接动机。

***

### JDK 25 紧凑对象头（JEP 519）

JDK 25 将 **Compact Object Headers**（JEP 519）提升为正式产品特性。该特性在 JDK 24 中曾作为实验性功能（JEP 450）引入，经过一年的打磨后在 JDK 25 中默认可用，无需额外解锁实验性参数。

**核心思想**：将 Mark Word 和 Class Pointer 合并为一个 64 位的紧凑头部，不再需要独立的类型指针字段。JVM 通过将类元数据地址编码进 Mark Word 的特定位段，在保持哈希码、锁状态、GC 标记等原有功能的同时，将对象头从 96 位（或 128 位）压缩到 64 位。

**标准对象头 vs 紧凑对象头**：

| 对比项 | 标准对象头（64 位 JVM） | 紧凑对象头（JEP 519） |
|--------|----------------------|---------------------|
| 总宽度 | 96 bit（Mark Word 64 + Class Pointer 32） | 64 bit（合并存储） |
| 普通对象大小 | 12 字节 | 8 字节 |
| 数组对象大小 | 16 字节 | 12 字节（长度字段仍需 4 字节） |
| 压缩指针上限 | 32 GB 堆内存 | 64 GB 堆内存 |
| 锁状态存储 | Mark Word 独立存储 | 嵌入紧凑头部的特定位段 |
| 启用参数 | 默认行为 | `-XX:+UseCompactObjectHeaders` |

**性能收益**（基于 Oracle 官方 JEP 519 测试数据）：

| 指标 | 提升幅度 | 测试基准 |
|------|---------|---------|
| 堆空间占用 | 减少约 22% | SPECjbb2015 |
| CPU 时间 | 减少约 8% | SPECjbb2015 |
| GC 次数 | 减少约 15% | 综合工作负载 |
| JSON 解析速度 | 提升约 10% | 特定 JSON Parser 基准 |

> **生产注意**：在 JDK 25 中启用紧凑对象头只需添加 `-XX:+UseCompactObjectHeaders`，不再需要 JDK 24 中的 `-XX:+UnlockExperimentalVMOptions`。但启用前建议在预发环境充分验证，尤其是 monitor 密集型的应用（如大量 `synchronized` 块或 `ReentrantLock` 使用场景），因为紧凑对象头的内部锁状态编码与标准对象头不同，极端竞争场景下可能存在未预见的性能回归。

**权衡与适用场景**：

- **优势显著**：小对象密集型应用（如缓存系统、消息队列、高频交易对象）收益最大；堆内存从 32 GB 扩展到 64 GB 时仍可保持压缩指针，避免指针膨胀带来的内存压力。
- **需谨慎验证**：依赖对象头精确布局的 Native 代码、使用 `Unsafe` 直接访问对象头的框架、以及 monitor 竞争极其激烈的场景，应在升级前进行压力测试。
- **与 ZGC / G1 的协同**：紧凑对象头减少的单对象内存 footprint 直接降低了 GC 的扫描和复制开销，与现代低延迟收集器的配合效果更佳。

### 实例数据与对齐填充

**实例数据**是对象真正存储的有效信息，即程序代码中定义的各种类型的字段内容。字段的排列顺序受虚拟机分配策略和字段在源码中的定义顺序影响。HotSpot 默认的分配策略为：相同宽度的字段总是被分配在一起（longs / doubles、ints、shorts / chars、bytes / booleans、oops），且父类中定义的变量会出现在子类变量之前。

字段排列策略可以通过 `-XX:FieldsAllocationStyle` 参数调整，但通常不需要修改。默认策略的目的是让相同宽度的字段连续存放，提高 CPU 缓存的利用率。

**对齐填充**（Padding）不是必然存在的，它仅仅起着占位符的作用。由于 HotSpot 要求**对象的大小必须是 8 字节的整数倍**，因此当对象头加上实例数据的大小不是 8 的倍数时，需要通过对齐填充来补全。

对齐填充看似简单，但在高并发场景下具有实际意义。CPU 缓存以**缓存行**（Cache Line，通常 64 字节）为单位加载数据。如果一个对象恰好跨越两个缓存行，那么在多核 CPU 上同时修改对象的不同字段时，可能会触发**伪共享**（False Sharing）问题——即两个线程修改的是不同变量，但因为它们位于同一缓存行，导致缓存频繁失效。

> 虽然对象对齐填充本身不直接解决伪共享问题（它只保证对象整体 8 字节对齐），但理解内存对齐的机制有助于我们在设计高并发数据结构时，通过手动填充（Padding）将热点字段隔离到不同的缓存行中，从而避免伪共享。JDK 8 引入的 `@Contended` 注解就是基于这一原理，自动在字段之间插入填充，将字段隔离到独立的缓存行中。

### 访问定位：句柄与直接指针

Java 程序通过栈上的 reference 数据类型来操作堆上的具体对象。对象的访问方式由虚拟机实现决定，主要有两种：

| 访问方式 | 原理 | 优点 | 缺点 |
|---------|------|------|------|
| **句柄访问** | 堆中划分句柄池，reference 存储句柄地址，句柄中包含对象实例数据和类型数据的各自地址 | 对象移动时（GC 回收的典型行为）只需修改句柄中的实例数据指针，reference 本身无需改动 | 访问对象需要两次指针定位，速度较慢 |
| **直接指针访问** | reference 直接存储对象地址 | 访问速度快，只需一次指针定位 | 对象移动时需要修改所有指向该对象的 reference |

**HotSpot 虚拟机使用直接指针访问方式**。这也是绝大多数现代 JVM 的选择，因为对象访问的频率远高于对象移动，牺牲移动时的开销来换取日常访问的性能是合理的权衡。

使用直接指针时，reference 直接指向对象头的起始地址。通过对象头中的类型指针，可以快速定位到类的元数据。这种设计使得对象访问非常高效，但在 GC 进行对象移动（如标记-整理算法中的整理阶段）时，需要暂停用户线程（Stop The World），更新所有指向被移动对象的 reference。

句柄访问方式虽然在对象移动时具有优势（只需修改句柄，无需遍历所有 reference），但由于对象访问是更频繁的操作，句柄方式的整体性能通常不如直接指针。历史上 Sun Classic VM 使用句柄方式，但现代 JVM 已经全部转向直接指针。

> 理解对象访问方式有助于理解为什么某些 GC 算法（如 CMS）不进行整理，从而避免对象移动的开销；而 G1、ZGC 等现代收集器虽然会移动对象，但通过并发整理和读屏障等技术，将对应用的影响降至最低。

## 生产环境 OOM 排查指南

OOM 是线上系统最常见的故障之一。不同区域的 OOM 有着不同的触发原因和排查方法，快速准确地定位问题区域是排障效率的关键。

### Java 堆溢出

`java.lang.OutOfMemoryError: Java heap space` 是最常见的 OOM 类型。触发原因通常有两类：

- **内存泄漏**：对象已经不再被使用，但由于某些引用（如静态集合、未注销的监听器、ThreadLocal 未清理）仍然存在，导致 GC 无法回收。内存泄漏的特点是堆内存持续增长，即使 GC 后也无法回落。
- **内存溢出**：对象确实都需要存活，但堆内存设置不足以支撑业务高峰期的使用量。这种情况通常表现为堆内存使用率在高峰期接近 100%，但 GC 后能够回落。

**排查思路**：

1. **开启堆转储**：添加 JVM 参数 `-XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/path/to/dump.hprof`，让 JVM 在 OOM 时自动保存内存快照。这是排查堆 OOM 最关键的一步，没有堆转储，很难精确定位问题。

2. **分析 dump 文件**：使用 MAT（Memory Analyzer Tool）或 VisualVM 分析 dump 文件：
   - 查看 Dominator Tree，找出占用内存最大的对象。
   - 使用 Leak Suspects 报告，MAT 会自动分析可能的内存泄漏点。
   - 如果是泄漏，追踪 GC Roots，找到持有泄漏对象的引用链，定位到具体的代码位置。

3. **区分泄漏和溢出**：
   - 如果大对象是业务必需（如缓存、大批量数据处理），则属于内存溢出，需要增加堆内存或优化数据处理方式。
   - 如果大对象是不应该存在的（如重复加载的缓存、未清理的会话数据），则属于内存泄漏，需要修复代码逻辑。

4. **优化策略**：
   - 对于内存泄漏：修复引用清理逻辑，使用 WeakReference、SoftReference 等引用类型，确保对象在不需要时能被 GC 回收。
   - 对于内存溢出：增加 `-Xmx`，优化算法减少内存占用，或引入分页、流式处理等机制避免一次性加载大量数据。

> 对于微服务架构，还需要关注是否存在某个接口返回了过大的对象列表，或者批量查询没有分页限制。一个常见的案例是：分页查询接口在参数校验缺失时，被调用方传入 `pageSize=999999`，导致一次性加载全表数据到内存。

### 虚拟机栈溢出

`java.lang.StackOverflowError` 和 `java.lang.OutOfMemoryError: unable to create new native thread` 都与虚拟机栈相关，但触发原因和排查方法完全不同。

**StackOverflowError**：

- **触发原因**：线程请求的栈深度超过虚拟机允许的最大深度。最常见的原因是无限递归或递归深度过大。另一个容易被忽视的场景是对象间的循环依赖导致的相互调用，比如两个对象的 `toString()` 方法相互引用。
- **典型场景**：
  - 深度优先搜索（DFS）未对递归深度做限制。
  - 复杂的对象关系图遍历，如 ORM 框架中双向关联对象未配置懒加载。
  - 正则表达式的灾难性回溯（Java 的正则引擎使用栈实现回溯）。
- **排查方法**：
  1. 查看堆栈跟踪的最后几行，通常就是递归调用的入口。
  2. 检查是否存在循环依赖，或者递归终止条件是否正确。
  3. 如果是正则表达式导致的问题，检查正则表达式是否存在回溯陷阱，考虑使用 possessive quantifier（如 `++`）或 atomic group（如 `(?>...)`）来避免回溯。

**unable to create new native thread**：

- **触发原因**：创建线程时操作系统无法分配足够的内存。HotSpot 中虚拟机栈和本地方法栈合并实现，不支持动态扩展，因此栈区域的 OOM 通常表现为无法创建新线程。
- **排查方法**：
  1. 使用 `jstack <pid>` 查看当前线程数量和线程状态，确认是否存在线程数量异常增长。
  2. 检查线程池配置，确认最大线程数是否合理。常见错误是将 `maximumPoolSize` 设置为 `Integer.MAX_VALUE`。
  3. 检查是否存在线程泄漏。例如，使用 `Executors.newFixedThreadPool` 创建的线程池，如果任务队列无限增长，线程池中的线程可能长期存活。
  4. 检查操作系统对进程线程数的限制（Linux 的 `ulimit -u` 和 `/proc/sys/kernel/threads-max`）。
  5. 适当减小 `-Xss` 参数，允许创建更多线程（注意权衡单个线程的栈深度）。

### 元空间溢出

`java.lang.OutOfMemoryError: Metaspace` 在 JDK 8 及以后版本中替代了原有的 PermGen OOM。

触发原因：

- **动态生成大量类**：CGLIB、Javassist、ASM 等字节码生成工具在运行时生成大量类。Spring AOP、MyBatis Mapper、Hibernate 代理等都是常见的触发源。
- **OSGi 或热部署**：模块动态加载和卸载时，如果类加载器未被正确释放，会导致类元数据持续累积。
- **反射使用过于频繁**：JVM 会为反射调用生成存取器类（Accessor），大量使用反射可能生成大量临时类。
- **Groovy / JRuby 等动态语言**：这些语言的运行时环境会动态生成大量类。

**排查思路**：

1. 使用 `jcmd <pid> VM.metaspace` 查看元空间使用详情，包括已用空间、 committed 空间和类加载器数量。
2. 使用 `jcmd <pid> GC.class_stats`（JDK 8u40+）或 `jmap -clstats <pid>` 查看类加载器统计信息，找出加载类最多的类加载器。
3. 检查代码中是否频繁使用动态代理或字节码生成，考虑复用生成的类或代理实例。
4. 如果确实需要加载大量类，设置合理的 `-XX:MaxMetaspaceSize`，并监控元空间增长趋势。
5. 对于热部署场景，确保旧的类加载器能够被 GC 回收。类加载器回收的条件是：该类加载器加载的所有类实例都已被回收、该加载器本身没有被引用。

> 一个典型的案例是 Spring Boot 应用配合大量 `@Configuration` 和 CGLIB 代理，在容器化部署时如果元空间上限设置过小（如默认的 128 MB），在应用启动或动态刷新上下文时容易触发 Metaspace OOM。解决方案通常是将 `-XX:MaxMetaspaceSize` 增大到 256 MB 或 512 MB，并监控实际使用量。

### 直接内存溢出

`java.lang.OutOfMemoryError: Direct buffer memory` 发生在直接内存区域。

触发原因：

- 大量使用 NIO 的 `ByteBuffer.allocateDirect()` 或 Netty 的 `Unpooled.directBuffer()`。
- `DirectByteBuffer` 的回收依赖 Full GC，如果分配速度快于回收速度，直接内存会被耗尽。
- 使用了支持零拷贝的框架（如 Netty、Kafka）但未正确配置直接内存大小。

**排查思路**：

1. 使用 `jcmd <pid> VM.native_memory summary` 查看 Native Memory 的使用情况，确认直接内存是否接近上限。
2. 检查代码中 `DirectByteBuffer` 的创建位置，是否存在创建后未合理释放的情况。
3. 考虑显式调用 `((DirectBuffer) buffer).cleaner().clean()` 来主动释放直接内存。但需要注意：过度使用可能干扰 GC，且需要处理 `cleaner()` 返回 null 的情况（某些 JDK 实现可能不支持）。
4. 增大 `-XX:MaxDirectMemorySize` 参数。对于 Netty 应用，通常建议将直接内存设置为与堆内存相当的大小。
5. 检查是否存在直接内存泄漏：某些框架可能缓存了 `DirectByteBuffer` 但未正确释放，导致直接内存持续增长。

### GC Overhead Limit Exceeded

`java.lang.OutOfMemoryError: GC overhead limit exceeded` 是一个特殊的 OOM。它的触发条件是：Java 进程花费 **98% 以上的时间执行 GC**，但只恢复了 **不到 2% 的堆内存**，且该情况连续重复了 5 次。

这个异常的本质是 JVM 认为继续执行 GC 已经没有意义，堆内存几乎完全由存活对象占据，无论怎么 GC 都无法释放足够的空间。与其让系统陷入无限 GC 循环，不如直接抛出 OOM 让应用失败。

**排查思路**：

1. 这个异常**通常意味着堆内存严重不足**，或者存在严重的内存泄漏。分析 GC 日志，确认是哪种 GC 算法、哪个代（Young / Old）在频繁回收。
2. 检查堆内存设置是否合理。如果 `-Xmx` 设置过小，而应用确实需要更多内存，增加堆内存是最直接的解决方案。
3. 检查是否存在内存泄漏。即使堆内存设置合理，如果存在泄漏，内存最终也会被耗尽，触发 GC overhead limit exceeded。
4. 在极少数情况下（如大量短期对象但不允许 OOM），可以通过 `-XX:-UseGCOverheadLimit` 关闭该检查，但**不推荐在生产环境使用**。关闭后，系统不会快速失败，而是陷入长时间的 GC 停顿，导致服务无响应，后果比快速失败更严重。

> GC overhead limit exceeded 和 Java heap space 的区别在于：前者意味着堆中还有少量空间，但 GC 效率极低；后者意味着堆中完全没有可用空间。前者通常发生在堆内存严重不足但尚未完全耗尽的阶段，后者发生在堆内存完全耗尽的阶段。两者的根因通常是相同的——内存泄漏或堆内存设置不足。

## JDK 版本演进要点

JVM 的内存结构并非一成不变。JDK 8 和 JDK 17 是两个重要的里程碑，带来了内存管理方面的重大调整。

### JDK 8 的关键调整

JDK 8 是 JVM 内存结构的分水岭，最核心的变化是**永久代的移除和元空间的引入**：

- **永久代被元空间取代**：方法区从 JVM 堆内存移入本地内存，彻底解决 `PermGen space` OOM 问题。这一改动使得动态类加载密集型的应用（如大量使用反射、动态代理、脚本引擎的系统）稳定性大幅提升。
- **字符串常量池移入 Java 堆**：原本位于永久代的字符串常量池被移入堆中。这意味着 `String.intern()` 的行为虽然不变，但 intern 后的字符串现在占用的是堆内存，需要关注堆的使用情况。对于大量使用 intern 的系统（如字典表、枚举值缓存），堆内存的计算需要纳入字符串常量池的占用。
- **静态变量移入 Java 堆**：类的静态变量从方法区（永久代）移入堆中。GC 可以更方便地管理这些引用，静态变量引用的对象现在受堆 GC 管理。

这些改动的直接效果是：动态类加载密集型的应用（如大量使用反射、动态代理、脚本引擎的系统）在 JDK 8 后稳定性大幅提升，不再频繁遭遇永久代溢出的困扰。同时，字符串常量池和静态变量的移入也使得堆内存的管理更加统一，简化了内存监控和调优的复杂度。

对于从 JDK 7 或更早版本升级到 JDK 8 的系统，需要注意：

- 不再需要设置 `-XX:MaxPermSize` 和 `-XX:PermSize`，取而代之的是 `-XX:MaxMetaspaceSize` 和 `-XX:MetaspaceSize`。
- 堆内存的监控需要纳入字符串常量池的占用。如果系统大量使用 `String.intern()`，升级后堆内存占用可能会增加。
- 类卸载的条件和时机有所变化。元空间的垃圾收集策略与永久代不同，某些场景下类卸载可能更积极，也可能更保守。

### JDK 17 的进一步优化

JDK 17 作为长期支持版本（LTS），在内存管理和并发性能方面做了多项优化：

- **ZGC 正式可用**：ZGC 是一款低延迟垃圾收集器，在 JDK 11 中作为实验特性引入，在 JDK 15 中正式发布，JDK 17 进一步完善。ZGC 通过**染色指针**（Colored Pointers）和**读屏障**（Load Barrier）实现并发整理，能够将 STW（Stop The World）时间控制在 10 毫秒以内。对于大堆内存（数百 GB）且对延迟敏感的场景（如金融交易系统、实时竞价系统），ZGC 是一个重要的选项。ZGC 不区分新生代和老年代，而是将所有对象统一管理，通过并发整理来回收内存。
- **移除偏向锁**（JDK 15 废弃，JDK 21 彻底移除）：如前所述，偏向锁在现代多线程应用中收益有限，反而带来维护开销。移除偏向锁简化了锁的状态转换流程，对于线程池密集型的应用可能有轻微性能提升。在 JDK 17 中，偏向锁默认已被禁用（`-XX:-UseBiasedLocking`），但代码仍然保留，直到 JDK 21 才被彻底移除。
- **增强的伪共享防护**：JDK 17 在标准库中进一步优化了数据结构的缓存行对齐，减少了多线程场景下的伪共享问题。`@Contended` 注解的支持也更加完善。
- **Foreign Function & Memory API**（孵化中，JDK 19 正式发布）：虽然 JDK 17 中该 API 仍处于孵化阶段，但它为直接内存的管理提供了更安全、更高效的替代方案，未来可能逐步替代 `DirectByteBuffer` 和 `Unsafe` 的部分功能。

> 对于生产环境的升级建议：如果当前运行的是 JDK 8，建议评估升级到 JDK 17 的可行性。升级带来的元空间稳定性、ZGC 低延迟特性以及新语言特性（如 Record、Pattern Matching、Sealed Class）能够显著提升系统的可维护性和性能天花板。但升级前需要进行充分的兼容性测试，特别是涉及反射、Unsafe、JNI 等底层 API 的代码。JDK 17 引入了强封装（Strong Encapsulation），默认不允许通过反射访问 JDK 内部类，这可能导致某些依赖内部 API 的库无法正常工作。

### Mark Word 的位级布局

Mark Word 是一个典型的**状态复用**数据结构——相同的 64 位空间，根据锁标志位的不同，承载着完全不同的信息。理解这一设计，首先要看清每一种锁状态下这 64 位是如何分配的。

#### 64 位 Mark Word 精确位布局

下面是从高 63 位到低 0 位的精确分配（HotSpot 64 位实现，little-endian 视角）：

| 锁状态 | bit 63~62 | bit 61~9 | bit 8~5 | bit 4 | bit 3 | bit 2 | bit 1~0 |
|--------|-----------|----------|---------|-------|-------|-------|---------|
| 无锁（未计算哈希码） | 未使用（61 bit） | — | 分代年龄（4 bit） | 0 | 偏向锁模式（0） | — | 01 |
| 无锁（已计算哈希码） | 未使用（25 bit） | 哈希码（31 bit） | 分代年龄（4 bit） | 0 | 偏向锁模式（0） | — | 01 |
| 偏向锁 | 偏向线程 ID（54 bit） | Epoch（2 bit） | 分代年龄（4 bit） | 未使用（1 bit） | 偏向锁模式（1） | — | 01 |
| 轻量级锁 | 指向栈中 Lock Record 的指针（62 bit） | — | — | — | — | — | 00 |
| 重量级锁 | 指向 ObjectMonitor 的指针（62 bit） | — | — | — | — | — | 10 |
| GC 标记 | 未使用（62 bit） | — | — | — | — | — | 11 |

> 上述布局对应的是 `biased_lock` 开启的情况（JDK 15 前默认），偏向锁模式位（bit 3）为 1 表示允许偏向，为 0 表示不允许。JDK 15 后该位始终为 0。

#### 哈希码的"延迟生成"

`Object.hashCode()` 的哈希码并不是在对象创建时就写入 Mark Word 的，而是在**第一次调用该方法（且未被重写）时才计算并存入**。计算方式通常涉及对象的内存地址（或 TLAB 分配顺序）与一个随机数混合，以降低碰撞概率。

这一设计带来了一个关键约束：**一旦对象的 `hashCode()` 被调用，该对象就永远无法进入偏向锁状态**。原因是哈希码（31 bit）和偏向线程 ID（54 bit）在 Mark Word 中占据的位置完全重叠——它们无法共存。JVM 的处理策略是：在计算哈希码时，如果对象当前处于偏向锁状态，则撤销偏向锁并膨胀为轻量级锁，然后将规范哈希码写入 Mark Word 的无锁布局中。

> **实战意义**：如果你将对象放入 `HashMap`（会调用 `hashCode()`），且该对象同时参与 `synchronized` 同步，那么这些同步块会直接从轻量级锁起步，跳过了偏向锁的优化阶段。在大量使用集合+同步的场景下，这是有意义的性能考量。

#### 分代年龄

分代年龄占 4 位，取值范围 0~15（`1111`），这与 `-XX:MaxTenuringThreshold` 的默认值 15 直接对应。每次 Minor GC 存活，年龄加 1；达到阈值后，对象晋升到老年代。这个字段在**所有锁状态下都保持不变**，是 Mark Word 中唯一一个不受锁状态切换影响的位段。

#### Epoch

偏向锁的 Epoch（2 bit）用于支持**批量重偏向**（Bulk Rebias）。当 JVM 发现某个类的对象频繁撤销偏向锁时，会通过递增该类全局 Epoch 并批量更新已偏向对象的 Epoch，来实现高效的批量撤销和重新偏向，避免逐个对象的 `stop-the-world` 安全点开销。

***

### 内存对齐的硬件原理

前文提到 HotSpot 要求对象起始地址对齐至 8 字节的倍数。这并非 JVM 的任意约定，而是**由硬件读写内存的物理方式决定的**。要理解为什么要对齐，首先需要了解内存的物理结构和 CPU 访问内存的过程。

#### 内存的物理结构

我们平时所称的"内存"也叫随机访问存储器（RAM），由插在主板扩展槽上的**存储器模块**（Memory Module）组成。每个存储器模块内部封装了 8 个 **DRAM 芯片**，编号 0~7。

每个 DRAM 芯片内部是一个二维矩阵，矩阵中的每个元素称为**超单元**（Supercell），大小为 1 字节（8 bit）。每个 supercell 通过行地址（RAS, Row Access Strobe）和列地址（CAS, Column Access Strobe）定位。

> 在程序员眼中连续的内存地址，在物理上其实是不连续的——连续的 8 个字节分别存储在不同的 DRAM 芯片上，每个芯片存储 1 个字节。

存储控制器通过 RAS 和 CAS 依次从 8 个 DRAM 芯片中读出 8 个字节，在控制器中聚合后返回给 CPU。这决定了**内存一次读取的最小单位是 8 字节**，在 64 位处理器中称为一个 **word size**。

#### 对齐访问与未对齐访问

当 CPU 请求读取一段连续内存时，如果起始地址是 8 的倍数（对齐），那么这段内存上的所有地址在 DRAM 芯片中对应的坐标（RAS, CAS）是相同的，CPU 可以通过**一次**读事务（read transaction）完成读取。

当起始地址不是 8 的倍数（未对齐），情况就不同了。比如读取 `0x0007` 到 `0x0014` 这 8 个字节：`0x0007` 的（RAS, CAS）与 `0x0008` 及之后的地址不同，CPU 被迫执行两次读事务：

1. 先从 `0x0000~0x0007` 读取 8 字节，左移提取 `0x0007`；
2. 再从 `0x0008~0x0015` 读取 8 字节，右移提取 `0x0008~0x0014`；
3. 将两次结果进行或运算，拼出完整的 8 字节。

一次对齐访问只需 1 次读事务，一次未对齐访问可能需要 2 次——理论上慢了一倍。

#### 原子性

CPU 可以原子地操作一个对齐的 word size 内存区域。如果数据跨越了两次内存 IO 的边界，CPU 就无法保证对该数据操作的原子性，这在并发场景下可能导致**读撕裂**（Torn Read）或**写撕裂**（Torn Write）。

#### 尽量落在一个缓存行内

CPU 高速缓存以**缓存行**（Cache Line）为单位存取数据，主流处理器中缓存行大小为 **64 字节**。如果对象起始地址对齐至 8 的倍数，对象就更可能完全落在一个缓存行内，避免跨缓存行存储导致的**多次缓存加载和一致性协议开销**。

> 这就是为什么 Java 对象必须对齐至 8 字节的倍数的三个根本原因：**CPU 访问速度**（减少读事务次数）、**操作原子性**（避免读/写撕裂）、**缓存利用率**（避免跨缓存行）。

### 压缩指针的位运算原理

将 32 位系统迁移到 64 位系统后，对象引用从 4 字节膨胀到 8 字节，类型指针也从 4 字节膨胀到 8 字节。这导致同样的对象，在 64 位系统中占用的内存几乎是 32 位系统的两倍，进而引发更频繁的 GC 和更低的 CPU 缓存命中率。

此时一个自然的想法是：能不能在 64 位系统中使用 32 位的对象引用，同时又拥有超过 4 GB 的寻址空间？压缩指针（Compressed OOPs）正是为此而生。

#### 核心机制：利用"浪费"的低 3 位

JVM 要求堆中对象的起始地址对齐至 8 字节的倍数（默认配置下）。这意味着对象地址的**二进制低 3 位永远为 `000`**——因为任何 8 的倍数，二进制形式的低 3 位都是 0。

既然这 3 位永远是 0，JVM 可以选择不在堆中存储它们。于是：

- **存储时**：将 64 位地址右移 3 位（去掉永远为 0 的低 3 位），得到 32 位的偏移量，存入堆中。
- **寻址时**：将 32 位偏移量左移 3 位（低 3 位补 0），加上基地址（堆的起始地址），恢复成完整的 64 位地址。

这相当于用 32 位存储，但寻址范围扩展到了 `2^32 × 8 = 2^35 = 32 GB`。

```
存储：64 位地址 >> 3 → 32 位偏移
寻址：32 位偏移 << 3 → 35 位有效地址空间
```

> 压缩指针默认在堆内存小于 32 GB 时自动开启（`-XX:+UseCompressedOops`）。当堆内存超过 32 GB 时，压缩指针自动关闭，对象引用回退到 64 位，内存占用显著增加。

#### 零基模式与非零基模式

上述描述对应的是**零基压缩**（Zero-based Compressed OOPs）模式，即堆的基地址为 `0`。在这种模式下，寻址计算最为简单，只需左移 3 位即可，无需额外加法运算。

当操作系统无法将 Java 堆映射到低地址空间（如 32 GB 堆在 64 位地址空间中恰好无法从 0 开始分配）时，JVM 会退化为**非零基模式**。非零基模式下，寻址需要先左移 3 位再加上一个非零的基地址（Narrow Oop Base）。这会增加一条加法指令，但性能影响很小，在绝大部分场景下可以忽略。

#### ObjectAlignmentInBytes 与寻址范围

对象对齐的倍数可以通过 JVM 参数 `-XX:ObjectAlignmentInBytes` 调整（默认 8，取值范围 8~256，必须是 2 的次幂）。对齐倍数越大，"浪费"的低位数越多，压缩指针能覆盖的寻址范围也越大：

| ObjectAlignmentInBytes | "浪费"的低位 bit 数 | 寻址范围 |
|-----------------------|---------------------|---------|
| 8（默认） | 3 bit | 32 GB |
| 16 | 4 bit | 64 GB |
| 32 | 5 bit | 128 GB |
| 64 | 6 bit | 256 GB |

> **不推荐随意增大此参数**：更大的对齐倍数意味着更粗粒度的对齐，对象之间的字节填充（Padding）也会相应增多，可能抵消压缩指针节省的空间。这个参数主要用于无法缩小堆内存却又需要压缩指针的特殊场景。

#### 为什么 ZGC 不支持压缩指针

ZGC 使用**染色指针**（Colored Pointers）技术，在 64 位指针中利用特定的 bit 位来存储 GC 元数据（如 Marked0 / Marked1、Remapped、Finalizable 标记）。ZGC 占用了指针的 **bit 0~3** 以及 **bit 44~47**。

而压缩指针正是利用低 3 位为 0 这一事实来扩充寻址空间——如果这些低 bit 位被 ZGC 用于染色，压缩指针的位运算前提就失效了。这就是为什么开启 ZGC 时，压缩指针会被自动禁用，且无法手动开启。ZGC 的大堆内存（TB 级）本身就是为不需要压缩指针的场景设计的。

### 伪共享与 @Contended

即使两个变量在逻辑上毫无关联，它们也可能因为被加载到同一个 CPU 缓存行中而互相拖累性能。这种问题在高并发系统中出现的频率远超预期。

#### 缓存行的工作方式

CPU 并非以字节为单位从内存读写数据，而是以**缓存行**（Cache Line）为单位，主流处理器的缓存行大小为 **64 字节**。即使你只读取一个 `long` 类型的变量（8 字节），CPU 也会将该变量所在的整个 64 字节缓存行加载到 L1/L2 缓存中。

同时，多核 CPU 之间通过**缓存一致性协议**（如 MESI / MOESI）来保证缓存数据的一致性。当一个核心修改了某个缓存行中的数据，其他核心中对应的缓存行会被标记为**失效**（Invalid），迫使它们在下次访问时重新从内存或共享缓存加载。

#### 伪共享的发生

假设一个类中有两个 `volatile long` 字段 `a` 和 `b`，它们在内存中相邻存储，共同被加载到同一个 64 字节缓存行中：

```java
class FalseSharding {
    volatile long a;
    volatile long b;
}
```

现在线程 A 反复修改字段 `a`，线程 B 反复修改字段 `b`。虽然 `a` 和 `b` 逻辑上互不相关，但因为它们共享同一个缓存行，任何一方的修改都会使对方的缓存行失效，导致双方反复从内存重新加载数据。这种现象就叫做**伪共享**（False Sharing）——本质上是在**共享缓存行，而不是共享数据**。

#### 性能影响

伪共享造成的影响与缓存一致性协议的开销直接相关。以 MESI 协议为例：

1. 核心 A 修改 `a` → 核心 B 的缓存行被置为 Invalid；
2. 核心 B 修改 `b` → 需要先加载完整的缓存行（64 字节），然后核心 A 的缓存行被置为 Invalid；
3. 核心 A 再次修改 `a` → 又需要重新加载缓存行……

每一次修改都需要跨核心的缓存同步操作，这些操作通常需要几十到上百个 CPU 周期。在高频写入的场景下（如计数器、队列指针的 CAS 更新），伪共享可能导致**吞吐量下降数倍甚至数十倍**。

#### @Contended 注解

JDK 8 引入了 `sun.misc.Contended` 注解（JDK 9 后移至 `jdk.internal.vm.annotation.Contended`），通过在字段前后插入**缓存行大小的填充**（Padding），确保被注解的字段独占一个缓存行，从而消除伪共享：

```java
class TrueSharding {
    @Contended
    volatile long a;
    @Contended
    volatile long b;
}
```

JVM 会在字段 `a` 和 `b` 的前后各填充 128 字节（前后各一个缓存行），使 `a` 和 `b` 分别落在不同的缓存行中。这样一来，线程 A 修改 `a` 不再影响线程 B 的 `b`。

> 默认情况下，`@Contended` 仅对 JDK 内部类生效。如果要在应用代码中使用，需要添加 JVM 参数 `-XX:-RestrictContended` 来解除限制。

#### 分组与 Padding 宽度

`@Contended` 支持**分组**（Group）机制。同一分组内的字段可以共享缓存行，但不同分组之间用填充隔开：

```java
class GroupedSharding {
    @Contended("group1")
    volatile long a;
    @Contended("group1")
    volatile long b;
    @Contended("group2")
    volatile long c;
    @Contended("group2")
    volatile long d;
}
```

在这里，`a` 和 `b` 共享 `group1` 的缓存行；`c` 和 `d` 共享 `group2` 的缓存行；两组之间以 128 字节填充隔离。

Padding 宽度的默认值可通过 `-XX:ContendedPaddingWidth` 参数调整（默认 128），但一般不需要修改，除非你使用的处理器缓存行大小不是标准的 64 字节。

#### 实际应用

JDK 标准库在多个关键位置使用了 `@Contended`，包括：

- `Thread` 类中的随机数种子字段（避免线程之间因伪共享拖慢并发随机数生成）；
- `ConcurrentHashMap` 的内部计数器（避免不同桶的计数器互斥）；
- `ForkJoinPool` 的任务队列（避免多个 worker 线程的任务窃取操作互相影响）；
- `LongAdder` / `Striped64` 的 Cell 数组（核心设计就是通过填充消除伪共享）。

> 在实际开发中，如果你需要维护一组被多个线程频繁更新的 `volatile` 长字段（如多维度计数器、滑动窗口统计），且性能测试显示吞吐量不随线程数线性扩展，就应当怀疑是否遭遇了伪共享。使用 `@Contended` 或手动填充 64 字节的 `long` 数组作为隔离块是最直接的解决手段。

## 总结

JVM 的内存结构是理解 Java 应用性能和行为的基础。从运行时数据区的职责划分，到对象在堆中的精确布局，再到各区域 OOM 的排查思路，这些内容构成了后端工程师排查内存问题的完整工具箱。

核心要点回顾：

- **线程私有区域**（程序计数器、虚拟机栈、本地方法栈）的生命周期与线程绑定，其中栈帧结构决定了方法调用的内存开销。局部变量表中的引用是重要的 GC Roots，理解这一点有助于分析对象为何未被回收。
- **线程共享区域**（堆、方法区、直接内存）是 OOM 的主要发生地，需要根据应用特性合理配置参数。TLAB 和逃逸分析是高并发场景下优化对象分配的关键机制。
- **对象头中的 Mark Word** 不仅存储哈希码和 GC 信息，更是 `synchronized` 锁升级过程的物理载体。理解锁状态位的变化，有助于在高并发场景下合理选择同步策略。
- **元空间替代永久代** 是 JDK 8 最重要的内存结构变革，解决了动态类加载的 OOM 痛点。字符串常量池和静态变量移入堆中，使得内存管理更加统一。
- 面对 OOM，关键是**先定位区域，再分析根因**，善用 `jcmd`、`jmap`、MAT 等工具进行诊断。不同区域的 OOM 排查思路截然不同，快速定位区域是排障效率的关键。
- **JDK 版本演进** 不断优化内存管理的效率和稳定性。从永久代到元空间的变革、ZGC 的低延迟特性、偏向锁的移除，都是为了适应现代应用的需求。

理解这些底层机制，不是为了在日常开发中手动管理内存，而是为了在系统出现内存问题时，能够快速、准确地定位根因，做出合理的参数调优或代码重构决策。内存问题的排查往往是一场与时间的赛跑，熟练掌握这些知识，能够让你在关键时刻快人一步。
