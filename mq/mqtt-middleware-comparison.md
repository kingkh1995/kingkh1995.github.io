## 背景与概述

物联网设备规模在过去十年经历了爆炸式增长。Gartner 预测 2026 年全球物联网连接数将突破 300 亿，这些设备产生的数据不再是简单的传感器读数，而是涉及车联网指令下发、工业 PLC 实时控制、智能家居状态同步等场景。传统 HTTP 短连接模型在海量设备长连接场景下暴露出诸多问题：连接开销大、服务端推送能力弱、协议头部冗余、实时性不足。MQTT 协议正是在这一背景下成为物联网消息传输的事实标准。

MQTT Broker（消息代理）处于物联网架构的枢纽位置，其核心职责包括：

- **连接管理**：维护数百万级设备的长连接，处理心跳保活与断线重连
- **消息路由**：基于 Topic 的通配符匹配规则，实现一对多、多对多的消息分发
- **会话管理**：持久化设备离线期间的消息，设备上线后自动恢复未消费消息
- **安全认证**：支持用户名/密码、Token、TLS 证书等多层认证机制
- **扩展性**：在设备规模增长时通过集群横向扩展承载更高的连接数与消息吞吐

市面上主流的 MQTT 中间件产品各具特色，本文将从架构设计和适用场景两个维度，对 EMQX、阿里云微消息队列 MQTT 版、Apache RocketMQ-MQTT、金蝶天燕 ApusicMQ（AMQ）和 RabbitMQ MQTT 插件进行对比分析。

## EMQX

### 产品定位

EMQX 是当前全球范围内应用最广泛的开源 MQTT Broker 之一，由杭州映云科技（EMQ）主导开发。它以高并发、低延迟的物联网消息接入能力著称，在车联网、工业物联网和智慧能源场景中占据领先地位。

### 开发语言与架构

EMQX 基于 Erlang/OTP 平台构建。Erlang 的 Actor 并发模型天生适合处理海量轻量级连接——每个 MQTT 连接映射为一个 Erlang 进程，进程间通过消息传递通信，无共享内存，避免了传统线程模型的锁竞争问题。

架构层面，EMQX 采用分布式集群设计：

- **无中心节点**：集群中每个节点角色对等，任意节点故障不影响整体可用性
- **数据分片**：订阅关系和会话信息通过 Mnesia 数据库分片存储，节点间自动同步
- **一致性保障**：采用 Ekka 库（基于 Raft 协议）实现集群元数据的一致性和自动选主
- **多层防火墙设计**：连接限制、速率控制、ACL 检查从接入层到路由层逐层过滤，防止异常流量穿透

### 核心性能指标

EMQX 官方测试数据显示：单节点可承载 200 万并发连接，集群环境下支持千万级设备接入，端到端消息延迟可控制在 5 毫秒以内。这些数据在实际部署中取决于硬件配置和业务消息体大小，但足以说明其架构对高并发场景的支撑能力。

### 版本体系

EMQX 提供开源版（Apache 2.0 许可）和企业版两种形态。开源版覆盖了 MQTT 5.0、QoS 0/1/2、遗嘱消息、保留消息、WebSocket 接入等核心功能。企业版在此基础上增加了规则引擎（SQL 方式处理消息）、数据桥接（流转发到 Kafka / 数据库 / HTTP 服务）、Kubernetes Operator 自动运维等功能。对于严苛的生产环境，企业版的数据集成能力可以显著降低开发成本。

### 适用场景

- 大规模设备接入场景，如千万级车联网平台
- 工业现场总线级别的实时控制，要求毫秒级延迟
- 需要规则引擎进行边缘预处理和数据流编排的场景
- 多协议支持需求（MQTT / MQTT-SN / CoAP / LwM2M / HTTP）

## 阿里云微消息队列 MQTT 版

### 产品定位

阿里云微消息队列 MQTT 版（简称 MQTT 云服务）是全托管的云原生 MQTT 消息服务。它与 RocketMQ 同属阿里云消息生态，目标是为物联网和移动互联网应用提供开箱即用的消息接入能力。

### 核心架构

MQTT 云服务的架构遵循典型的"无状态网关 + 有状态存储"分层设计：

**网关层**由分布式无状态节点组成，负责设备长连接的接入和协议处理。网关节点的核心职责包括：

- 处理 MQTT CONNECT 请求，完成设备身份认证与权限校验
- 维护设备心跳（Keep Alive），超时未收到 PINGREQ 时主动断开连接并触发遗嘱消息
- 将 MQTT Topic 映射为 RocketMQ 的 Topic，完成协议转换与消息转发
- 支持 WebSocket 接入，方便浏览器端与移动端集成

**存储层**依赖消息队列 RocketMQ 作为后端存储，解耦了连接管理与会话持久化。消息流转路径如下：设备发布消息 → 网关接收 → 转存至 RocketMQ Topic → 匹配订阅关系的消费者拉取消息 → 网关将消息 Push 给订阅设备。

这种架构设计的优势在于：

- 网关层无状态，可以按需水平扩展以承载更高的设备并发
- 消息的持久化、重试、死信等复杂逻辑由 RocketMQ 兜底，降低网关的复杂度
- 设备端与云端（RocketMQ 应用）天然互通，业务系统不需要额外的协议转换层

### 关键特性

- **高并发能力**：千万级在线设备连接，百万级消息并发处理
- **设备级权限控制**：支持基于设备的 Topic 发布/订阅权限隔离，防止设备越权访问
- **SSL/TLS 加密**：全链路加密传输，满足金融级安全合规要求
- **消息轨迹**：提供消息全生命周期的追踪能力，方便排查消息丢失或重复的问题
- **一对多广播**：支持设备维度的广播消息下发，适用于固件升级和设备配置下发场景

### 消息互通模型

MQTT 云服务与 RocketMQ 的互通模型是其区别于其他 MQTT Broker 的显著特性。以车联网场景为例：车载终端通过 MQTT 上报 GPS 位置信息，数据存储在 RocketMQ 的 `gps_report` Topic 中；后端定位服务（一个 RocketMQ 消费者）从 Topic 拉取数据做轨迹分析，同时计算结果也可以通过 RocketMQ 消息直接转为 MQTT 消息推送给指定车辆。整个链路不需要额外的桥接组件，端到端延迟可控。

### P2P 点对点消息

P2P（点对点）消息支持设备间通过唯一标识直接通信，无需经过 Topic 广播。这对智能家居场景中手机 App 直接控制单个设备的场景非常实用，减少了订阅关系的维护成本。

### 适用场景

- 已深度使用阿里云 RocketMQ 的团队，需要 IoT 设备接入能力
- 车联网、智能家居、移动推送等需要端-云双向通信的场景
- 希望减少运维成本、直接使用托管服务的团队

## Apache RocketMQ-MQTT

### 项目定位

Apache RocketMQ-MQTT（项目代号 RocketMQ-5.0-MQTT 扩展）是在 RocketMQ 5.x 架构之上构建的 MQTT 5.0 协议网关实现。它不是 RocketMQ 的插件，而是 RocketMQ 多协议架构的一个组成部分，旨在实现 IoT 设备消息与服务端消息的统一接入和存储。

### 核心架构：协议计算与存储分离

RocketMQ-MQTT 沿用了 RocketMQ 5.x 的"计算与存储分离"设计理念，分为两层：

**协议计算层（MQTT Gateway）** 是一个可插拔部署的独立组件。它负责：

- MQTT 协议的完整解析与编码，管理设备连接的生命周期（CONNECT / SUBSCRIBE / PUBLISH / PINGREQ / DISCONNECT）
- 服务端 Topic 匹配，将 MQTT Topic 通配符（+ / #）转换为 RocketMQ 的消息过滤表达式
- 订阅关系的管理与缓存，以及设备在线状态管理
- 与 RocketMQ Broker 通过 gRPC 通信，发送 PUBLISH 消息和接收订阅反馈

**Broker 层** 专注于消息存储和分发。MQTT 消息写入后与其他协议（Remoting / gRPC）共享同一个 CommitLog，通过 ConsumerQueue 多级索引实现不同协议的消费分发。

### 队列存储模型

RocketMQ-MQTT 的设计精髓在于存储层面的复用。一条 MQTT 消息写入 CommitLog 后，根据消息的属性（Topic、QueueId、Tag）产生多个维度的索引：

- 对于普通 RocketMQ 消费者，通过标准的 ConsumerQueue 索引消费
- 对于 MQTT 设备消费者，通过 MQTT 专用的 ConsumerQueue（或基于消息过滤）实现按 Topic 通配符匹配的消费

这种"一份数据，多场景复用"的模式避免了消息在 MQTT 网关和 RocketMQ 之间复制两份导致的一致性问题。传统方案中，消息需要先写入 IoT 网关自身的存储，再转发一份到消息队列给后端应用消费，两份数据的一致性保障非常困难。RocketMQ-MQTT 依靠统一存储直接从源头消除了这一矛盾。

### 推拉结合的消费模型

MQTT 协议原生是 Push 模型——服务端将消息推送给客户端。但 RocketMQ 底层存储是 Pull 模型。RocketMQ-MQTT 采用了一个精巧的推拉结合策略来调和二者：

1. RocketMQ Broker 在有新消息到达时通过 Notify 机制向 MQTT 网关发送轻量级事件通知
2. 网关收到通知后，根据设备订阅的 Topic 列表判断消息是否匹配
3. 若匹配，网关主动向 Broker 发起 Pull 请求拉取消息体
4. 网关将拉取到的消息通过 MQTT PUBLISH 推送至终端设备

这一模型既保持了 RocketMQ 存储层的 Pull 优势（批量消费、顺序控制），又向终端设备呈现了 Push 体验。

### 适用场景

- 已有 RocketMQ 基础设施，需引入 MQTT 设备接入能力的团队
- 应用端和 IoT 设备端需要共享同一份消息数据的场景
- 关注数据一致性、不希望引入跨系统数据同步的架构

## 金蝶天燕 ApusicMQ（AMQ）

### 产品定位

ApusicMQ（AMQ）是金蝶天燕自主研发的企业级消息中间件，属于信创国产化替代体系中的关键组件。它全面支持 JMS 1.1 和 JMS 2.0 规范，在政府、金融、军工等对自主可控有严格要求的行业有较高渗透率。

### 架构特性

AMQ 支持多种消息模式：

- **P2P（点对点）**：基于 Queue 模型，一条消息仅被一个消费者消费
- **Pub-Sub（发布订阅）**：基于 Topic 模型，一条消息被所有订阅者消费

部署方面，AMQ 支持单机模式和集群模式，集群模式下采用主备方式保障高可用。安全方面支持 SSL/TLS 加密传输、ACL 权限控制、消息加密存储等能力。客户端支持 Java、C++、C# 等主流语言。

### MQTT 协议支持

AMQ 的核心定位是企业级 JMS 消息中间件，MQTT 协议支持并非其主要发力方向。当前版本对 MQTT 的支持停留在基础能力层面：支持基本的 MQTT 3.1.1 协议接入，能够完成 Topic 的发布/订阅，但在大规模设备连接管理、QoS 2 支持、物联网场景的集群弹性扩展等方面与专用 MQTT Broker 存在差距。

### 适用场景

- 政府、金融、国防等信创国产化政策驱动场景
- 已有金蝶中间件产品体系，需要消息中间件实现内部系统集成
- 对 JMS 规范有强依赖的传统企业应用

## RabbitMQ MQTT 插件

### 实现原理

RabbitMQ 通过 `rabbitmq_mqtt` 插件获得 MQTT 协议支持。该插件将 MQTT 协议桥接到 RabbitMQ 的 AMQP 协议体系，工作原理如下：

- 每个 MQTT 客户端连接在 RabbitMQ 内部对应一个 AMQP 连接
- MQTT Topic 订阅映射为 AMQP Topic Exchange 上的绑定关系
- 每个 MQTT 订阅对应一个内部 AMQP 队列，设备发布的消息经过 Exchange 路由后投递到对应队列
- 插件负责 MQTT 协议的解析与 AMQP 协议的互转

### 功能特性

RabbitMQ MQTT 插件支持 MQTT 3.1.1 和 5.0 协议（不同版本插件覆盖范围不同），关键功能包括：

- QoS 0 和 QoS 1 支持
- TLS / SSL 加密传输
- OAuth 2.0 和 JWT 认证
- 遗嘱消息（Last Will and Testament）
- WebSocket 桥接（通过 Web-MQTT 插件）
- 与 AMQP 客户端的消息互通

### 限制与不足

通过协议桥接实现 MQTT 支持必然带来额外开销：

- **协议转换开销**：每一条 MQTT 消息都需要经过 MQTT ↔ AMQP 的协议转换，占用额外的 CPU 和内存资源
- **连接密度受限**：每个 MQTT 连接在内部会创建对应的 AMQP 资源，单机承载的设备连接数远低于 EMQX 等专用 Broker
- **QoS 2 支持有限**：大部分版本的 RabbitMQ MQTT 插件对 QoS 2（仅一次送达）的支持不完整
- **订阅膨胀问题**：大量 MQTT 客户端订阅通配符 Topic 时，AMQP 内部的队列和绑定数量急剧膨胀，对管理平面造成压力

### 适用场景

- 团队已深度使用 RabbitMQ 处理业务消息，需要快速接入少量 MQTT 设备
- IoT 设备规模较小（千级以内），且对延迟和吞吐要求不苛刻
- 需要 MQTT 设备与现有 AMQP 消费者应用实现直接消息互通的原型验证

## 总结对比

### 对比表

| 维度 | EMQX | 阿里云 MQTT | RocketMQ-MQTT | ApusicMQ (AMQ) | RabbitMQ MQTT |
|------|------|-------------|---------------|----------------|---------------|
| 开发语言 | Erlang/OTP | Java（网关）+ RocketMQ | Java（RocketMQ + 网关） | Java | Erlang |
| 协议支持 | MQTT 3.1/3.1.1/5.0 | MQTT 3.1/3.1.1/5.0 + WebSocket | MQTT 5.0 | MQTT 3.1.1 | MQTT 3.1.1/5.0 |
| 架构类型 | 分布式原生 MQTT 集群 | 无状态网关 + RocketMQ 存储 | 协议计算层 + RocketMQ 存储 | 主备集群 | AMQP 协议桥接 |
| 最大连接数 | 单节点 200 万，集群千万级 | 千万级（托管） | 取决于 RocketMQ 集群规模 | 万级 | 十万级 |
| 消息存储 | 内置（Mnesia / 磁盘文件） | RocketMQ 持久化 | RocketMQ CommitLog | 磁盘文件 | AMQP Queue |
| QoS 支持 | 0/1/2 | 0/1/2 | 0/1/2 | 0/1 | 0/1（2 不完整） |
| 部署方式 | 自建 / K8s / EMQX Cloud | 全托管 | 自建（RocketMQ 集群 + 网关） | 自建 | 自建（RabbitMQ + 插件） |
| 运维复杂度 | 中（需维护 Erlang 集群） | 低（全托管） | 高（需管理 RocketMQ 集群 + 网关） | 中 | 低（RabbitMQ 生态成熟） |
| 许可证 | 开源版 Apache 2.0 / 企业版商用 | 商用收费 | Apache 2.0 | 商用 | MPL 2.0 |

### 选型建议

- **纯 IoT 场景，设备规模大，追求极致性能**：选择 EMQX。其 Erlang 架构在连接密度和实时性方面具有天然优势，开源版即可支撑百万级设备接入。
- **阿里云基础设施已有 RocketMQ 业务**：选择阿里云微消息队列 MQTT 版。端-云消息互通模型可以减少架构复杂度，全托管也免去了运维负担。
- **自建 RocketMQ，需要统一 IoT 与服务端消息**：选择 RocketMQ-MQTT 扩展。一份数据存储在 RocketMQ 中，同时供 MQTT 设备和后端应用消费，避免跨系统的一致性问题。
- **信创合规要求**：选择 ApusicMQ（AMQ）。虽然 MQTT 能力有限，但满足国产化认证是这类场景的首要考量。
- **已有 RabbitMQ，少量 MQTT 设备接入需求**：开启 `rabbitmq_mqtt` 插件即可。不需要引入新的中间件系统，适合快速原型和低规模场景。
